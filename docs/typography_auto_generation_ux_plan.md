## 0. 요약

현재 패널은 **파라미터를 먼저 보여주고 의도를 나중에 묻는** 구조다. 이 기획안은 순서를 뒤집는다: 편집자가 "어떤 호흡으로 읽히길 원하는가"를 먼저 고르면, 실제 가사가 어떻게 잘리는지 즉시 보여주고, 생성 전에 읽기 밀도와 위험 구간을 수치로 확인시키며, 생성 직전에만 파괴적 범위(교체 대상 개수)를 명시적으로 확인받는다.

세 가지 핵심 변경:

1. **의도 우선 화면** — 리듬 프리셋 3종을 첫 의사결정으로 올리고, 대표 가사 라인의 실시간 분할 샘플을 함께 보여준다.
2. **위험 중심 미리보기** — 카드 나열 대신 읽기 속도(자/초), 최단 노출 시간, 주의/위험 카운트를 앞세우고, 옵션 변경 시 자동 재계산한다.
3. **생성 시점 안전장치** — 이전 생성물 개수와 교체/유지 선택을 생성 버튼 직전에 보여주고, 파괴적 정리 기능은 생성 이후 영역으로 분리한다.

## 1. 설계 원칙

요청서의 판단 기준("기능이 많아 보이는 패널"보다 "편집자가 실전에서 덜 불안한 패널")에서 도출한 원칙. 이후 모든 결정은 이 원칙을 우선순위 순서대로 적용한다.

| # | 원칙 | 의미 |
|---|------|------|
| P1 | **결과를 먼저 보여준다** | 파라미터 숫자보다 "이 가사가 이렇게 잘린다"가 먼저. 모든 옵션 변경은 즉시 미리보기에 반영된다. |
| P2 | **파괴는 항상 예고한다** | 레이어를 만들거나 지우기 전에 대상 개수와 범위를 보여준다. "Everyric이 만든 것만 건드린다"를 매번 명시한다. |
| P3 | **초안임을 숨기지 않는다** | 생성물은 편집용 초안이다. 완성본처럼 보이게 하는 시각 요소(정교한 WYSIWYG 캔버스 등)는 오히려 지양한다. |
| P4 | **반복 루프를 짧게 만든다** | 조정 → 확인 → 재생성 한 사이클이 클릭 3회 이내여야 한다. 전문가의 반복 작업이 1차 사용자다. |
| P5 | **상태를 침묵시키지 않는다** | 진행·취소·부분 성공을 모두 문구로 노출한다. 조용한 무효화(현재: 옵션 변경 시 계획이 소리 없이 사라짐)는 금지. |

## 2. 접근 방식 검토

세 가지 UX 아키텍처를 비교했다.

### A안 — 단계형 위저드
데이터 확인 → 의도 선택 → 미리보기 → 생성을 화면 단위 스텝으로 강제.

- 장점: 초심자 안내가 확실하고 불안이 가장 적다.
- 단점: 반복 작업마다 스텝을 다시 밟아야 해서 P4에 정면으로 어긋난다. 전문 편집자에게 답답하다.

### B안 — 단일 화면 + 계층적 노출 (권장)
현재의 단일 탭 구조를 유지하되, 정보 순서를 의도 → 결과 → 세부 조정으로 재배열하고 세부 파라미터는 접는다.

- 장점: 반복 루프가 가장 짧다. 좁은 CEP 패널 폭에 맞다. 기존 구현 재사용률이 높다.
- 단점: 첫 사용 시 학습이 필요하다 → 분할 샘플과 위험 지표가 이를 보완한다.

### C안 — 미리보기 캔버스 중심
화면 배치를 시각적으로 렌더링하는 미리보기가 주인공이 되는 구조.

- 장점: 예측성이 가장 높아 보인다.
- 단점: 패널 폭 제약으로 실효성이 낮고, 구현 비용이 크며, "완성형 디자인을 만들어주는 도구"라는 잘못된 인상을 줘서 P3(비목표: 완성형 템플릿)와 충돌한다.

**결론: B안을 기본 골격으로, C안의 위험 시각화 요소(밀도 스트립)를 미리보기에 흡수하고, A안의 확인 스텝은 파괴적 지점(생성 실행) 한 곳에만 적용한다.**

## 3. 핵심 질문에 대한 답

요청서의 7개 질문에 대한 결론. 근거는 각 산출물 섹션에 있다.

1. **첫 화면은 어떤 의사결정부터 받아야 하는가?**
   "읽기 리듬"(분할 밀도)이다. 레이아웃·글자 크기가 아니다. 최악의 실패("한 글자씩 나오는 결과")가 분할 밀도에서 나오고, 장르·취향 편차가 가장 큰 축도 분할 밀도이기 때문이다. 데이터 준비 여부는 의사결정이 아니라 상태 표시로 처리한다(없으면 배지 + 싱크 탭 이동 버튼).

2. **Readable / Balanced / Rhythmic 밀도 프리셋은 충분한가?**
   1차 축으로는 충분하며, 장르 프리셋보다 낫다. 행동 기술형 이름(어떻게 잘리는지)은 결과와 일치하지만, 장르 이름(발라드·힙합)은 같은 장르 안에서도 편집 스타일이 갈리므로 배신당한 느낌을 준다. 장르는 프리셋의 **설명 태그**로만 쓴다("Rhythmic — 랩·빠른 훅에 적합"). 장르 프리셋 팩(리듬+레이아웃+여운 조합)은 차기 버전에서 별도 계층으로 얹는다.

3. **분할 기준을 어떤 언어로 보여줘야 하는가?**
   숫자가 아니라 **실물 예시**다. 로드된 가사의 대표 라인 1~2개를 현재 설정으로 잘라서 "이렇게 잘립니다"를 실시간으로 보여준다. 파라미터 라벨도 편집자 어휘로 바꾼다: PHRASE SIZE → "블록 글자 수", PAUSE CUT → "쉼 분할 기준(이 이상 쉬면 나눔)", POST-ROLL → "여운 유지". 섹션 6.2 라벨 표 참조.

4. **생성 전 미리보기는 어느 수준까지?**
   3단 구성: ① 요약 지표(카드·레이어 수, 평균 읽기 속도, 최단 노출) ② 위험 구간 목록(주의/위험 판정 카드) ③ 카드별 분할 텍스트. 픽셀 단위 배치 렌더링은 하지 않는다 — 디자인은 생성 대상이 아니므로(P3) 배치 정보는 프리셋 이름과 정렬 방향 아이콘 수준이면 충분하다. 밀도 스트립(시간축 시각화)은 차기.

5. **재생성 루프는 어떻게 설계하는가?**
   계획 계산은 순수 함수이므로(현재 planner.ts 구조 그대로) 옵션 변경 시 **자동 재계산**(300ms 디바운스)한다. "미리보기 버튼을 다시 눌러야 하는지" 고민 자체를 제거한다. 생성 후에는 같은 버튼이 "다시 생성 (이전 96개 교체)"로 바뀌어 루프가 이어진다. 카드 단위 개입(합침/나눔/제외)과 구간 재생성은 차기.

6. **위험 기능(마커 제거, 생성물 정리)은 어디에 둬야 하는가?**
   생성 CTA와 물리적으로 분리한 "정리 도구" 접이식 섹션에 모은다. 모든 항목에 범위 라벨을 붙인다("Everyric이 만든 것만 지웁니다 · 사용자 레이어는 안전"). "이전 생성물 교체"는 설정 서랍에 숨기지 않고 생성 확인 단계에서 매번 개수와 함께 보여준다.

7. **자주 만지는 옵션 vs 숨겨도 되는 옵션?**
   - 항상 노출(Tier 1): 리듬 프리셋, 블록 글자 수, 등장 방식(reveal), 레이아웃, 기본 크기 — 곡마다·시안마다 바뀐다.
   - 접힘(Tier 2): 쉼 분할 기준, 블록당 최대 단어, 카드당 블록 수, 프리롤/포스트롤 — 취향 결정 후 프로젝트 내에서 고정된다.
   - 설정 서랍(Tier 3): 레이어 순서, 교체 기본값, Python 경로 — 환경 설정이며 곡 단위로 바뀌지 않는다.

## 4. 산출물 1 — 사용자 여정

페르소나: 뮤직비디오·리릭 비디오를 만드는 모션 디자이너. AE 숙련자, 이 패널은 처음 또는 반복 사용자.

```
[준비]          [의도 선택]        [검증]             [생성]            [다듬기·반복]
싱크 데이터  →  리듬 프리셋 선택 →  위험 지표 확인  →  교체 범위 확인  →  AE에서 편집
JSON/로컬 싱크   분할 샘플 확인      주의 카드 조정      생성 실행          마커 참조
                                                                        ↺ 재생성 루프
```

### 단계별 상세

**1. 준비 — "재료가 있는가"**
- 진입 시 데이터 배지가 상태를 말해준다: `NO DATA` → 안내 + "싱크 탭으로" 버튼 / `yorushika.json · 42라인` → 바로 진행.
- 불안 요소: "내 데이터가 제대로 읽혔나" → 라인 수·원자 수·길이 요약이 이미 싱크 탭에 있으므로 타이포 탭에는 출처 배지만 재노출.

**2. 의도 선택 — "어떤 호흡으로 읽히게 할 것인가"**
- 리듬 프리셋 3종을 카드형 선택지로 제시. 선택 즉시 대표 가사 라인의 분할 샘플이 갱신된다.
- 대표 라인 선정: 글자 수 기준 중앙값 라인 1개 + 최장 라인 1개(최악 사례를 미리 보여줌).
- 불안 요소: "프리셋이 내 곡에 맞나" → 샘플이 실제 가사이므로 즉시 판단 가능.

**3. 검증 — "이대로 만들면 사고가 없는가"**
- 미리보기는 버튼 없이 자동 갱신. 요약 지표(카드·레이어·평균 자/초·최단 노출)와 주의/위험 카운트가 상단에 고정.
- 위험 카드를 열면 어느 가사가 왜 위험한지(노출 0.3초, 11자/초)와 완화 방향("블록 글자 수를 낮추거나 쉼 기준을 줄이세요")이 보인다.
- 불안 요소: "레이어 몇백 개가 쏟아지면 어쩌지" → 레이어 수가 생성 전에 항상 보인다.

**4. 생성 — "무엇이 만들어지고 무엇이 지워지는가"**
- 생성 버튼 → 인라인 확인 스텝: 생성될 레이어 수, 발견된 이전 Everyric 생성물 개수, 교체/유지 선택, "사용자 레이어·마커는 건드리지 않습니다" 고지, Undo 1회 복원 고지.
- 실행 중: 진행 문구 + 취소 버튼. 완료: 생성/교체/건너뜀 개수 보고.

**5. 다듬기·반복 — "AE에서 이어서 작업"**
- 타임라인에서 카드 단위 라벨 색·이름 규칙으로 구조가 보인다(섹션 9).
- 라인 마커를 참고로 켜고 편집. 마음에 안 들면 패널로 돌아와 옵션 조정 → "다시 생성 (96개 교체)" → 이전 생성물만 정리되고 새로 생성.
- 이 루프가 전문 사용자의 주 사용 패턴이며, 클릭 수는 조정 1~n회 + 생성 1회 + 확인 1회다.

## 6. 산출물 3 — 옵션 정보 구조와 기본값

### 6.1 노출 계층

| 계층 | 위치 | 항목 | 근거 |
|------|------|------|------|
| Tier 1 | 항상 노출 | 리듬 프리셋, 블록 글자 수, 등장 방식, 레이아웃, 기본 크기 | 곡·시안 단위로 바뀌는 창작 결정 |
| Tier 2 | "세부 조정" 접힘 | 쉼 분할 기준, 블록당 최대 단어, 카드당 블록 수, 미리 등장, 여운 유지 | 취향 확정 후 프로젝트 내 고정 |
| Tier 3 | 설정 서랍 | 레이어 순서, 교체 기본값, Python 경로, 엔진/언어 | 환경 설정, 곡 단위로 안 바뀜 |

### 6.2 라벨 및 기본값

| 사용자 라벨 (제안) | 현재 라벨 | 내부 키 | 기본값 | 설명 문구 (제안) |
|---|---|---|---|---|
| 읽기 리듬 | DENSITY | `density` | balanced | 아래 값들을 한 번에 설정하는 프리셋 |
| 블록 글자 수 | PHRASE SIZE | `phraseTargetChars` | 9자 | 한 번에 읽는 덩어리 크기 |
| 블록당 최대 단어 | WORDS / BLOCK | `maxTokensPerBlock` | 4 | 이보다 많으면 블록을 나눕니다 |
| 쉼 분할 기준 | PAUSE CUT | `pauseThreshold` | 0.32s | 이 이상 쉬면 새 블록으로 나눕니다 |
| 등장 방식 | REVEAL | `revealMode` | 구문 싱크 | 구문 싱크: 부를 때 등장 / 함께: 줄 시작에 등장 |
| 미리 등장 | PRE-ROLL | `preRollFrames` | 3fr | 소리보다 살짝 먼저 보여줍니다 |
| 여운 유지 | POST-ROLL | `postRollFrames` | 8fr | 끝난 뒤에도 잠시 남겨둡니다 |
| 카드당 블록 수 | BLOCKS / CARD | `maxBlocksPerCard` | 4 | 한 화면에 담는 최대 문장량 |
| 레이아웃 | LAYOUT | `layout` | Auto | 카드 안 블록 배치 방식 |
| 기본 크기 | BASE SIZE | `fontSize` | 94px | 긴 문장은 자동으로 줄어듭니다 |
| 레이어 순서 | LAYER ORDER | `layerOrder` | 시간이 위로 | 타임라인 스택 방향 |

기본값은 현재 구현값을 유지한다(검증된 출발점). 변경되는 것은 노출 순서와 언어다.

### 6.3 동작 규칙

- 옵션 변경 → 300ms 디바운스 후 계획 자동 재계산 → 요약 지표 갱신. "미리보기 버튼" 개념 제거(P5: 조용한 무효화 금지).
- 프리셋이 시드한 값을 직접 수정하면 프리셋 표시가 "Balanced · 수정됨"으로 바뀐다. 프리셋을 다시 클릭하면 시드값으로 복귀.
- 계산 실패(빈 데이터 등) 시 요약 영역에 원인 문구를 표시하고 생성 버튼을 비활성화한다.

## 7. 산출물 4 — 프리셋 체계

### 7.1 구조: 2계층

```
리듬 프리셋 (1차 축, 행동 기술형) ── Readable / Balanced / Rhythmic
        │  시드: 블록 글자 수 · 최대 단어 · 쉼 기준
        ▼
현재 값 (수정 가능, "수정됨" 표시) ── 마지막 사용값 자동 저장 (localStorage, 현행 유지)
```

### 7.2 리듬 프리셋 정의 (시드값은 현행 유지)

| 프리셋 | 시드 | 설명 카피 | 장르 태그 (설명용) |
|---|---|---|---|
| Readable | 12자 · 5단어 · 쉼 0.48s | 문장 호흡으로 크게 묶습니다 | 발라드 · OST · 느린 곡 |
| Balanced | 9자 · 4단어 · 쉼 0.32s | 대부분의 곡에 맞는 권장 기본 | 팝 · 록 |
| Rhythmic | 6자 · 3단어 · 쉼 0.20s | 박자를 따라 빠르게 나눕니다 | 랩 · 댄스 · 빠른 훅 |

장르는 **프리셋 이름이 아니라 설명 태그**다. 근거: 섹션 3-2.

### 7.3 확장 (차기)

- **사용자 정의 프리셋**: 현재 값 묶음을 이름 붙여 저장("내 리릭비디오 스타일"). Tier 1~2 값 전체를 캡처.
- **장르 프리셋 팩**: 리듬 + 레이아웃 + 여운 조합의 상위 번들(예: "발라드 팩" = Readable + Center + 여운 16fr). 리듬 프리셋 위에 얹는 별도 계층으로, 1차 축을 대체하지 않는다.
- **언어별 기본 보정**: 일본어(글자당 정보량 높음)는 블록 글자 수 -2, 영어는 단어 기준 가중 등. 싱크 데이터의 `language` 필드로 자동 제안.

## 8. 산출물 5 — 생성 전 미리보기 UX

### 8.1 3단 구성

| 단 | 내용 | 갱신 시점 |
|---|------|----------|
| 요약 지표 | 카드 수 · 레이어 수 · 평균 읽기 속도(자/초) · 최단 노출 시간 | 옵션 변경 시 자동 |
| 위험 목록 | 주의/위험 판정 카드 + 원인 + 완화 방향 문구 | 자동 |
| 카드 상세 | 카드별 시간·분할 텍스트·레이아웃, 위험 카드 인라인 표시 | 버튼으로 펼침 |

### 8.2 위험 판정 기준 (제안 초기값, 내부 상수로 조정 가능)

| 판정 | 조건 | 표시 |
|---|---|---|
| 🔴 위험 | 블록 노출 < 0.4s 또는 읽기 속도 > 10자/초 | 빨강 + 카드 ID + 원인 + 완화 문구 |
| ⚠ 주의 | 블록 노출 < 0.8s 또는 읽기 속도 > 7자/초 | 노랑 + 카드 ID |
| 레이어 과다 | 총 레이어 > 250 (현행 경고 유지) | 요약 영역 경고 배너 |

읽기 속도 = 블록 글자 수 ÷ 화면 노출 시간. 노출 시간은 pre/post-roll 반영 후 기준. 판정은 블록 단위로 계산하되, 사용자에게는 카드 단위로 묶어 보고한다(편집자가 개입하는 단위가 카드이므로).

완화 문구는 원인별 고정 매핑(추측성 문구 금지):
- 노출 부족 → "쉼 분할 기준을 낮추면 앞뒤 블록과 합쳐집니다."
- 속도 초과 → "블록 글자 수를 줄이거나 Rhythmic 프리셋을 시도해보세요."

### 8.3 하지 않는 것

- 픽셀 단위 배치 렌더링(WYSIWYG 캔버스). 디자인은 이 도구의 산출물이 아니며(비목표), 완성본 착시를 만든다(P3). 배치는 레이아웃 프리셋 이름 + 정렬 방향 텍스트로만 표기.
- 밀도 스트립(시간축 히트맵)은 가치는 있으나 MVP에서 제외(섹션 11).

## 9. 산출물 6 — 생성 후 타임라인 정리 UX

### 9.1 레이어 이름 규칙

```
EV2 C01-B02 · 너의 목소리가
└┬┘ └──┬──┘   └────┬────┘
그룹  카드-블록   텍스트 앞 12자
```

- 결정적 이름 + 레이어 코멘트 메타데이터(현행 유지)로 교체 시 자기 생성물만 식별.
- 텍스트 스니펫이 이름에 있어야 편집자가 타임라인에서 가사를 스크럽 없이 찾는다.

### 9.2 시각 구조화

- **라벨 색 순환**: 카드 단위로 AE 라벨 컬러를 순환 적용(예: 8색 사이클). 타임라인에서 카드 경계가 색 띠로 보인다. 구현 비용 대비 효과가 가장 큰 항목.
- **레이어 순서**: 기본 "시간이 위로"(bottom-to-top, 현행 기본) 유지. AE 편집자의 스택 관성과 일치.

### 9.3 참고 마커

- 라인 마커는 컴포지션 마커 대신 **전용 잠금 널 레이어("EV2 · Timing Guide")**에 부착하는 방식을 권장. 근거: PRD의 "composition and layer marker tracks remain clean" 원칙, 사용자 컴프 마커와의 충돌 원천 차단, 제거가 레이어 삭제 1회로 끝남.
  - ※ 현재 구현(`everyricCreateLineMarkers`)이 컴프 마커 방식이라면 마이그레이션 항목. 구현 확인 필요.
- 마커 코멘트: `L07 · 고요한 밤 너의…` (라인 번호 + 텍스트).

### 9.4 정리 기능 배치

| 기능 | 위치 | 범위 라벨 |
|---|---|---|
| 이번 생성물만 제거 | 정리 도구 (접힘) | "방금 생성한 그룹(groupId)만 제거합니다" |
| 모든 Everyric 마커 제거 | 정리 도구 (접힘) | "Everyric 마커만 제거 · 사용자 마커는 안전" |
| 이전 생성물 교체 | 생성 확인 스텝 | 발견 개수와 함께 매번 표시 (설정 서랍에서 기본값만 설정) |

모든 정리 버튼은 실행 전 대상 개수를 버튼 라벨에 표시한다: "이번 생성물만 제거 (96)".

| N7 | 마커 전용 가이드 레이어 이전 | 현행 마커 방식 확인 후 (9.3) |

## 12. 산출물 9 — 실제 곡 테스트 시나리오 5개

각 시나리오는 프로파일이 다른 실패 모드를 겨냥한다.

### T1 — 느린 한국어 발라드 (예: 아이유 '밤편지' 류)
- 프로파일: 템포 60~70, 긴 모음 지속, 라인 사이 간주 김.
- 절차: Readable 선택 → 샘플 확인 → 생성.
- 검증: 긴 모음 구간에서 블록이 과분할되지 않는가(쉼 기준 0.48s 동작), 간주 구간이 빈 카드 없이 스킵되는가, 여운 유지가 다음 라인과 겹치지 않는가.
- 예상 위험: post-roll이 다음 카드 시작을 침범 → 카드 겹침 경고 필요 여부 확인.

### T2 — 고밀도 보카로이드 곡 (예: DECO*27 'ラビットホール' 류)
- 프로파일: BPM 140+, 글자 단위 atom(일본어), 훅 반복, 밀도 극단.
- 절차: Balanced로 시작 → 위험 다수 발생 확인 → Rhythmic 전환 → 위험 감소 확인 → 생성.
- 검증: 위험 판정이 실제 "읽기 어려운 구간"과 일치하는가, 완화 문구를 따르면 실제로 위험이 줄어드는가(재생성 루프의 실효성), 레이어 250개 경고 동작.
- 예상 위험: 글자 단위 atom에서 4자 청크 분할(planner의 단일 토큰 경로)이 어색한 절단을 만드는지.

### T3 — 영어 팝 (예: Dua Lipa 'Levitating' 류)
- 프로파일: 워드 단위 atom, 긴 단어 혼재, 싱코페이션.
- 절차: Balanced → 블록당 최대 단어 조정 반복 → 생성.
- 검증: 글자 수 기준과 단어 수 기준이 충돌할 때(긴 단어 1개 > 9자) 분할이 자연스러운가, 영어 구두점 분할 동작.
- 예상 위험: 평균 자/초 지표가 영어(단어 인지 단위)에서 과민 판정 → 언어별 임계 보정(N8) 근거 수집.

### T4 — 한/영 혼합 K-pop (예: 뉴진스 'Super Shy' 류)
- 프로파일: 한 라인 안에서 언어 전환, 훅 반복 다수, 빠른 챈트.
- 절차: Rhythmic → 생성 → 훅 반복 구간 비교.
- 검증: 언어 전환 지점에서 분할 일관성, 동일 훅이 반복마다 같은 분할을 얻는가(결정성), 카드 ID·라벨 색으로 반복 구간이 타임라인에서 식별되는가.
- 예상 위험: 혼합 스크립트에서 글자 수 계산의 인지 밀도 편차.

### T5 — 라이브 음원 · 저신뢰 싱크 (예: 밴드 라이브 부틀렉)
- 프로파일: confidence 낮음, 반주/애드립 구간, 타이밍 드리프트.
- 절차: 로컬 싱크 생성 → 낮은 신뢰 경고 확인 → 라인 마커 추가 → AE에서 마커로 검수 → 부분 조정 후 재생성.
- 검증: 낮은 confidence 라인이 미리보기에서 식별되는가(차기 후보), 마커 기반 검수 흐름이 실제로 쓰이는가, 취소·부분 성공 문구 동작.
- 예상 위험: 드리프트 구간에서 pre/post-roll이 오히려 어긋남을 증폭.

**공통 판정 기준**: 각 시나리오에서 (a) 첫 생성까지의 클릭 수 ≤ 5, (b) 위험 지표를 보고 조정 → 재생성 루프 1회가 1분 이내, (c) 생성 후 Undo 1회 완전 복원, (d) 사용자 레이어·마커 무손상.
