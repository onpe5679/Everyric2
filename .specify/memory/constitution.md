# Everyric2 Constitution

## Core Principles

### I. User Empowerment Over Platform Lock-in

**어떤 노래든, 어디서든, 누구나 싱크된 가사를 볼 수 있어야 한다.**

기존 서비스는 DB에 있는 노래만 지원하고, 신곡/인디/커버는 포기하게 만든다. Everyric2는 이 한계를 AI로 돌파한다. 사용자가 원하면 **직접 생성**할 수 있어야 하고, 그 결과는 **커뮤니티에 환원**되어야 한다.

- 플랫폼이 지원 안 하면 우리가 만든다
- 한 사람이 생성하면 모두가 혜택을 받는다
- 오픈 데이터(LRCLIB)와 협력한다

### II. Privacy as Default

**사용자의 오디오와 가사 원문은 절대 저장하지 않는다.**

처리에 필요한 데이터는 처리 직후 삭제한다. 저장하는 것은 오직 **타임스탬프**(시작/끝 시간)와 **가사 해시**뿐이다. 이는 법적 리스크 회피가 아니라, 사용자 신뢰를 지키기 위한 원칙이다.

- 오디오: 처리 후 즉시 삭제
- 가사 원문: 저장 안 함, 해시만 저장
- 사용자 식별 정보: 수집 안 함

### III. Engine Agnosticism

**특정 엔진에 종속되지 않는다.**

CTC, WhisperX, SOFA, MFA 등 다양한 엔진이 존재하며, 각각 장단점이 있다. torchaudio의 forced_align처럼 deprecated 되는 것도 있다. 엔진은 언제든 교체될 수 있어야 하며, 새 엔진 추가가 기존 코드를 건드리지 않아야 한다.

- 모든 엔진은 동일한 인터페이스를 따른다
- 엔진별 의존성은 선택적 설치로 분리한다
- 엔진 선택은 사용자에게 맡긴다

### IV. Local-First, Cloud-Optional

**로컬에서 완전히 동작해야 한다. 클라우드는 편의 옵션이다.**

인터넷 없이도, API 키 없이도, 핵심 기능이 동작해야 한다. 클라우드는 "설치 없이 쓰고 싶은 사람"을 위한 것이지, 필수가 아니다. 이는 사용자 자율성과 오프라인 사용성을 보장한다.

- CLI는 완전히 오프라인으로 동작
- 클라우드 의존 기능은 명확히 표시
- 로컬 LLM(Ollama 등) 지원

### V. Multi-Platform Consistency

**Chrome, After Effects, CLI에서 동일한 결과를 얻어야 한다.**

세 개의 프로그램이 각각 다른 결과를 내면 안 된다. 동일한 오디오와 가사를 넣으면 동일한 타임스탬프가 나와야 한다. 이를 위해 모든 클라이언트는 **동일한 엔진**을 사용하거나, **동일한 API**를 호출한다.

- 공통 데이터 모델 사용
- API 응답 포맷 통일
- 클라이언트는 UI만 담당, 로직은 서버/CLI에 위임

## Non-Goals

명확히 **하지 않는 것**:

- 가사 저작권 데이터베이스 구축 (LRCLIB과 협력)
- 음원 다운로드 기능 제공 (Chrome Store 정책 위반)
- 모든 언어 지원 (우선순위: 일본어 > 영어 > 한국어 > 중국어)
- 실시간 스트리밍 처리 (녹음된 오디오만 지원)

## Governance

### Amendment Process

1. Constitution 변경은 심사숙고 후에만 제안
2. 변경 시 **기존 원칙과의 충돌 여부** 검토 필수
3. "왜 이 원칙이 필요한가"에 대한 명확한 이유 필요

### Version Policy

- **MAJOR**: 핵심 원칙 제거 또는 근본적 변경
- **MINOR**: 새 원칙 추가 또는 명확화
- **PATCH**: 문구 수정, 오타 교정

### Compliance

모든 기능/변경은 이 헌법의 원칙을 준수하는지 자문해야 한다:

- 이 변경이 사용자 자율성을 침해하는가?
- 이 변경이 특정 엔진/플랫폼에 종속을 만드는가?
- 이 변경이 프라이버시 원칙을 위반하는가?
